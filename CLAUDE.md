# STM32F407 AIoT Bring-up Project Rules

This is an STM32F407ZGT6 project generated by STM32CubeMX for an STM32 + FreeRTOS + AIoT desktop environment assistant.

## Editing Boundaries

Do not edit these CubeMX-generated or toolchain files unless the user explicitly asks:

- `stm32f407_aiot_bringup.ioc`
- `Core/Src/main.c` outside `USER CODE BEGIN` / `USER CODE END` blocks
- `Core/Inc/main.h` outside `USER CODE BEGIN` / `USER CODE END` blocks
- `Core/Src/stm32f4xx_hal_msp.c`
- `Core/Src/stm32f4xx_it.c`
- `Core/Inc/stm32f4xx_it.h`
- `Core/Src/system_stm32f4xx.c`
- `Core/Inc/stm32f4xx_hal_conf.h`
- `startup_stm32f407xx.s`
- `STM32F407XX_FLASH.ld`
- `Drivers/**`
- `Middlewares/**`
- `cmake/stm32cubemx/**`

Peripheral configuration, clock tree, pin muxing, middleware enablement, and FreeRTOS settings should be changed in STM32CubeMX first, then code should be regenerated.

## Preferred Edit Locations

Put application code in these locations:

- `App/**`
- `Core/Src/app_*.c`
- `Core/Inc/app_*.h`
- `Core/Src/main.c` only inside `USER CODE BEGIN` / `USER CODE END` blocks
- `Core/Inc/main.h` only inside `USER CODE BEGIN` / `USER CODE END` blocks

## Required Workflow

Before editing protected files:

1. Explain why the protected file must be changed.
2. Ask the user for confirmation.
3. Keep the change as small as possible.

Before and after code changes, check:

```sh
git status --short
git diff
```

Do not delete generated STM32CubeMX files, startup files, linker scripts, HAL drivers, CMSIS files, or build configuration files.

## Project Direction

The application layer should focus on:

- FreeRTOS task structure
- UART logging
- GPIO LED/button bring-up
- I2C sensors and OLED
- WiFi module communication
- MQTT state reporting
- watchdog, reconnect, and fault reporting logic
