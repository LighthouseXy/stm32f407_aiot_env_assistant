# STM32F407 AIoT 桌面环境助手项目功能介绍

## 项目介绍

本项目是一个基于 STM32F407 + FreeRTOS 的桌面环境监测助手。项目以正点原子探索者 STM32F407ZGT6 开发板为主控，结合 AHT30 温湿度传感器、板载光敏传感器、OLED、ESP01S WiFi 模块和本地按键，实现环境数据采集、本地显示、串口日志、WiFi 入网和后续 MQTT 扩展。

项目定位偏智能硬件 / AIoT / 桌面设备方向，不追求复杂工业控制，而是强调嵌入式项目的完整闭环：传感器采集、状态管理、本地显示、按键交互、串口调试和联网能力。

```text
环境采集 -> 统一状态管理 -> OLED/串口显示 -> 按键交互 -> WiFi 入网 -> MQTT 扩展
```

## 已完成的功能

### 本地环境采集

- AHT30 通过 `I2C1 PB6/PB7` 接入，地址为 `0x38`。
- 板载光敏传感器使用 `PF7 / ADC3_IN5` 真实采样。
- 温湿度、光照百分比、传感器状态统一写入 `AppState_t`。
- 串口日志中可以看到 AHT30 状态、温湿度、ADC 原始值和光照百分比。

实现价值：

- 同时覆盖 I2C 数字传感器和 ADC 模拟量采样。
- 通过 `light_raw` 和 `light_percent` 分离原始数据与显示数据，便于后续做滤波和阈值判断。
- 传感器状态写入统一状态结构，为 OLED、串口和 MQTT 上报复用同一份数据打基础。

### OLED 与串口显示

- OLED 接入 `I2C1`，地址为 `0x3C`。
- OLED 显示温度、湿度、光照百分比和 WiFi 状态。
- USART1 输出系统状态、传感器数据、按键动作和 WiFi 入网日志。
- OLED 和串口共用同一份 `AppState_t`，避免显示层和日志层状态不一致。

实现价值：

- OLED 提供本地可视化状态，串口保留完整调试信息。
- 温湿度显示前先转成字符串，减少对 `printf` 浮点格式化的依赖。
- WiFi 状态使用短文本显示，适配 128x64 OLED 的显示空间。

### 按键交互

- KEY0：切换 `NORMAL / FOCUS / ALERT`。
- KEY1：打印系统状态，并触发 I2C 扫描。
- KEY2：清除报警状态，必要时回到 `NORMAL`。
- 按键逻辑包含软件防抖和长按防重复触发。

实现价值：

- KEY0 / KEY1 / KEY2 分别承担模式切换、诊断输出、报警恢复，职责清晰。
- 使用独立按键状态变量，避免多个按键之间互相影响。
- I2C 扫描挂到 KEY1，方便现场排查 AHT30 和 OLED 是否在线。

### ESP01S WiFi 入网

- ESP01S 使用 `USART3 PB10/PB11` 通信。
- 已完成 `AT / ATE0 / AT+CWMODE_CUR=1 / AT+CWJAP / AT+CIFSR`。
- WiFi 状态已写入 `AppState_t`。
- OLED 第四行支持 `W:OFF / W:CONN / W:OK / W:ERR`。

实现价值：

- 从最小 AT 通信推进到完整 WiFi 入网流程。
- 串口日志和 OLED 显示都能反映 WiFi 状态，便于判断联网阶段。
- `AppState_t` 已预留 MQTT 状态字段，后续可以继续扩展云端上报。

### 工程结构

- 应用层代码集中放在 `App/` 目录。
- `app_state` 负责统一系统状态。
- `app_sensor` 负责 AHT30 和光敏采样。
- `app_display` / `app_oled` 负责 OLED 显示。
- `app_wifi` 负责 ESP01S AT 指令和 WiFi 入网流程。
- `app_mqtt` 保留 MQTT 扩展入口。

实现价值：

- 应用逻辑与 CubeMX 生成代码尽量分离，降低后续重新生成工程时的维护成本。
- 传感器、显示、按键、WiFi 之间通过统一状态结构解耦。
- MQTT 模块先保留扩展入口，避免后续新增联网功能时重新调整整体结构。

## 问题与解决

### OLED 刷新链路

问题现象：

```text
OLED 初始化代码存在，但屏幕没有实时显示预期内容。
```

定位过程：

不能只看初始化是否调用成功，需要区分三段链路：

```text
AppDisplay_Init()      -> OLED 初始化
App_DisplayTaskLoop()  -> OLED 周期刷新
App_SensorTaskLoop()   -> AppState 数据更新
```

解决思路：

1. `displayTask` / `sensorTask` 是否创建并运行。
2. `App_DisplayTaskLoop()` 是否进入 `AppDisplay_Update()`。
3. `AppState_t` 是否有最新数据。
4. I2C 连接、地址和硬件本身。

结果：

```text
明确了 OLED 问题不应只归因于硬件或 I2C 地址，任务循环和状态更新链路同样需要检查。
```

### ESP01S AT 接收

问题现象：

```text
ESP01S 模块通过 CH340G 单独测试可返回 OK，但 STM32 侧一度只能看到回显开头 A。
```

定位过程：

- ESP01S 模块本身正常。
- `USART3 TX/RX` 硬件链路可用。
- 主要问题来自临时 AT 接收逻辑：发送后接收窗口太短，日志边收边打导致观察结果不完整。

解决方案：

```text
发送 AT 指令
等待接收窗口
先缓存完整响应
最后统一打印 text 和 hex
```

结果：

```text
STM32 侧可以稳定看到 AT 响应，并支撑 ATE0、CWMODE、CWJAP、CIFSR 入网流程。
```

### AHT30 与 OLED 共用 I2C1

问题背景：

```text
AHT30 和 OLED 都挂在 I2C1 上，如果后续任务并发访问，总线访问需要统一调度。
```

当前处理：

- 先完成单任务周期下的采集和显示验证。
- 通过 KEY1 触发 I2C 扫描，确认 `0x38` 和 `0x3C` 两个设备在线。

后续优化：

```text
引入 I2C mutex 或统一 I2C 访问入口，避免多任务并发访问同一总线。
```
