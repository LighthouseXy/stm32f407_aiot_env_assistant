# STM32F407 AIoT 桌面环境助手：设计与实现

## 项目介绍

本项目是一个基于 STM32F407 + FreeRTOS 的桌面环境监测终端。系统以正点原子探索者 STM32F407ZGT6 开发板为主控，结合 AHT30 温湿度传感器、板载光敏传感器、OLED、ESP01S WiFi 模块和本地按键，实现环境数据采集、本地显示、串口诊断、WiFi 入网和 MQTT 环境数据上报。

系统面向智能硬件、AIoT 与桌面设备场景，围绕传感器采集、统一状态管理、本地交互、运行诊断和云端通信建立完整数据链路。

```text
环境采集 -> 统一状态管理 -> OLED/串口输出 -> 按键交互 -> WiFi 入网 -> MQTT 数据发布
```

## 功能设计

### 本地环境采集

- AHT30 通过 `I2C1 PB6/PB7` 接入，地址为 `0x38`。
- 板载光敏传感器使用 `PF7 / ADC3_IN5` 真实采样。
- 温湿度、光照百分比、传感器状态统一写入 `AppState_t`。
- 串口日志中可以看到 AHT30 状态、温湿度、ADC 原始值和光照百分比。

实现价值：

- 同时覆盖 I2C 数字传感器和 ADC 模拟量采样。
- 通过 `light_raw` 和 `light_percent` 分离原始数据与显示数据，为滤波和阈值判断保留清晰的数据边界。
- 传感器状态写入统一状态结构，为 OLED、串口和 MQTT 上报复用同一份数据打基础。

### OLED 与串口显示

- OLED 接入 `I2C1`，地址为 `0x3C`。
- OLED 显示温度、湿度、光照百分比和 WiFi 状态。
- USART1 输出系统状态、传感器数据、按键动作和 WiFi 入网日志。
- OLED 和串口共用同一份 `AppState_t`，避免显示层和日志层状态不一致。

实现价值：

- OLED 提供本地可视化状态，串口输出完整运行与通信信息。
- 温湿度显示前先转成字符串，减少对 `printf` 浮点格式化的依赖。
- WiFi 状态使用短文本显示，适配 128x64 OLED 的显示空间。

### 按键交互

- KEY0：切换 `NORMAL / FOCUS / ALERT`。
- KEY1：打印系统状态，并触发 I2C 扫描。
- KEY2：清除报警状态，必要时回到 `NORMAL`。
- 按键逻辑包含软件防抖和长按防重复触发。

实现价值：

- KEY0 / KEY1 / KEY2 分别承担模式切换、诊断输出、报警恢复，职责清晰。
- 使用独立按键状态变量，避免多个按键之间互相影响。
- I2C 扫描挂到 KEY1，方便现场排查 AHT30 和 OLED 是否在线。

### ESP01S WiFi 入网

- ESP01S 使用 `USART3 PB10/PB11` 通信。
- 已完成 `AT / ATE0 / AT+CWMODE_CUR=1 / AT+CWJAP / AT+CIFSR`。
- WiFi 状态已写入 `AppState_t`。
- OLED 第四行支持 `W:OFF / W:CONN / W:OK / W:ERR`。

实现价值：

- AT 指令层覆盖模块探测、回显关闭、Station 模式配置、热点接入和 IP 查询。
- 串口日志和 OLED 显示同步反映 WiFi 状态，能够定位当前联网阶段。
- `AppState_t` 统一管理 WiFi 与 MQTT 状态，为本地显示和云端通信提供一致的数据源。

### MQTT 环境数据发布

- 使用 ESP01S AT 指令建立到 `broker.hivemq.com:1883` 的 TCP 连接。
- 在 STM32 侧构造 MQTT 3.1.1 CONNECT 报文。
- 通过 `AT+CIPSEND` 发送 MQTT CONNECT 和 PUBLISH 二进制报文。
- 串口日志可观察到 `SEND OK`、`+IPD,4:` 和 CONNACK 成功报文 `20 02 00 00`。
- 发布 topic 为 `stm32f407/env`，payload 包含温度、湿度、光照百分比和当前模式。
- HiveMQ WebSocket Client 可订阅并收到 STM32 发布的 JSON 数据。
- MQTT 状态写入 `AppState_t`，可区分 `OFF / CONNECTING / CONNECTED / ERROR`。

实现价值：

- 建立从 WiFi 入网、TCP 连接、MQTT 握手到平台接收 JSON 数据的端到端链路。
- CONNECT 报文由 STM32 本地构造，覆盖 MQTT 固定头、协议名、协议等级、连接标志、Keep Alive 和 Client ID。
- PUBLISH 报文由 STM32 本地构造，平台端可直接查看 JSON 环境数据。
- 串口同时打印 text 和 hex，便于确认不可见的 MQTT 二进制响应和 ESP01S AT 回复。

### 工程结构

- 应用层代码集中放在 `App/` 目录。
- `app_state` 负责统一系统状态。
- `app_sensor` 负责 AHT30 和光敏采样。
- `app_display` / `app_oled` 负责 OLED 显示。
- `app_wifi` 负责 ESP01S AT 指令和 WiFi 入网流程。
- `app_mqtt` 负责 MQTT CONNECT / PUBLISH 报文构造、发送和结果判断。

实现价值：

- 应用逻辑与 CubeMX 生成代码尽量分离，降低后续重新生成工程时的维护成本。
- 传感器、显示、按键、WiFi 之间通过统一状态结构解耦。
- MQTT 模块封装 CONNECT / PUBLISH 报文构造和 TCP 发送流程，可复用于后续周期发布策略。

## 问题与解决

### OLED 刷新链路

问题现象：

```text
OLED 初始化代码存在，但屏幕没有实时显示预期内容。
```

定位过程：

不能只看初始化是否调用成功，需要区分三段链路：

```text
AppDisplay_Init()      -> OLED 初始化
App_DisplayTaskLoop()  -> OLED 周期刷新
App_SensorTaskLoop()   -> AppState 数据更新
```

解决思路：

1. `displayTask` / `sensorTask` 是否创建并运行。
2. `App_DisplayTaskLoop()` 是否进入 `AppDisplay_Update()`。
3. `AppState_t` 是否有最新数据。
4. I2C 连接、地址和硬件本身。

结果：

```text
明确了 OLED 问题不应只归因于硬件或 I2C 地址，任务循环和状态更新链路同样需要检查。
```

### ESP01S AT 接收

问题现象：

```text
ESP01S 模块通过 CH340G 单独测试可返回 OK，但 STM32 侧一度只能看到回显开头 A。
```

定位过程：

- ESP01S 模块本身正常。
- `USART3 TX/RX` 硬件链路可用。
- 主要问题来自临时 AT 接收逻辑：发送后接收窗口太短，日志边收边打导致观察结果不完整。

解决方案：

```text
发送 AT 指令
等待接收窗口
先缓存完整响应
最后统一打印 text 和 hex
```

结果：

```text
STM32 侧可以稳定看到 AT 响应，并支撑 ATE0、CWMODE、CWJAP、CIFSR 入网流程。
```

### AHT30 与 OLED 共用 I2C1

设计约束：

```text
AHT30 和 OLED 共用 I2C1；当多个任务并发访问总线时，需要通过互斥或统一入口进行调度。
```

当前策略：

- 当前采集与显示按照既定任务周期访问总线。
- 通过 KEY1 触发 I2C 扫描，确认 `0x38` 和 `0x3C` 两个设备在线。

扩展策略：

```text
引入 I2C mutex 或统一 I2C 访问入口，避免多任务并发访问同一总线。
```

### MQTT 调试链路

问题现象：

```text
WiFi 已经获取 IP，但 MQTT 阶段一度出现 AT 超时、CIPMUX 超时、TCP 连接后 CLOSED、CONNACK timeout。
```

定位过程：

- WiFi 入网成功不代表 ESP01S 立即可稳定响应下一条 AT 指令。
- `AT+CIPSTART` 成功后，如果等待过久才发送 MQTT CONNECT，broker 可能主动关闭 TCP 连接。
- `SEND OK` 和 `+IPD,4: 20 02 00 00` 可能在同一段串口回复中返回，不能把二者完全拆成两个独立接收窗口。
- MQTT 报文缓存和接收缓存如果放在任务栈上，会增加 FreeRTOS 任务栈压力。

解决方案：

```text
TCP 连接前先 AT 探测
CIPMUX 偶发失败时等待下一轮重试
收到 AT 期望关键字后尽快退出接收窗口
发送 payload 后继续接收 SEND OK 后的 +IPD
大缓存改为静态区
```

结果：

```text
[mqtt] tcp connect OK
[esp] tcp payload SEND OK
[mqtt] CONNECT packet SEND OK
[mqtt] CONNACK OK
[mqtt] PUBLISH packet SEND OK
```

### MQTT PUBLISH 调试链路

问题现象：

```text
CONNECT / CONNACK 已成功，但 PUBLISH 阶段一度出现 SEND OK 尾部漏采、CIPSEND 提示符超时、CONNACK 数据补收不完整、浮点字段为空。
```

定位过程：

- ESP01S 会返回 `Recv xx bytes`、`SEND OK` 和 `+IPD`，这些内容可能被拆成多段进入 UART 接收窗口。
- `CONNACK` 是二进制数据，不能只按可见字符串判断。
- STM32 默认 `snprintf` 对浮点格式化支持有限，直接使用 `%.1f` 会导致 JSON 中 `temp` / `hum` 为空。

解决方案：

```text
TCP payload 发送成功判定兼容 SEND OK 尾部和 Recv xx bytes
CONNACK 不完整时进入补收状态
CONNACK 后延迟一轮再发送 PUBLISH
温湿度转为整数小数位拼接，避免依赖浮点 printf
```

结果：

```text
[mqtt] CONNACK OK
[mqtt] PUBLISH packet SEND OK
{"temp":27.6,"hum":51.2,"light":97,"mode":"NORMAL"}
```

## 演进路线

### 本地交互增强

1. 优化 OLED 信息层级和中文界面。
2. 增加启动页、设备状态页和报警页。
3. 将 MQTT 状态纳入 OLED 统一状态展示。

### 运行能力增强

1. 增加 WiFi 失败重试与断线恢复。
2. 为 I2C1 共用总线增加互斥或统一访问调度。
3. 增加 MQTT 周期发布、连接保持与异常恢复。

### 智能环境助手

1. 扩展 CO2、空气质量和人体存在感知。
2. 增加 RGB 状态提示、环境报警和自动通风控制。
3. 接入语音交互和云端大模型服务，实现环境问答、状态解释和设备控制。
